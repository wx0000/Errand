# P5 — Stabilization + Delivery (repo-build slice)

## Context

The suite is functionally complete: **8 flows (4 green + 4 red-findings)**, **5 subflows**, all
**R1–R5 covered**, every finding human-adjudicated, P4 pushed to `origin/main`. What is missing is
the **reviewer-facing delivery layer** and a **stability + hygiene pass** so the suite comes up
clean from a fresh clone on the reviewer's machine (macOS / M2 / Android ARM64 emulator over `adb`).

P5 adds **no new flows**. It is documentation + verification + hygiene only:
- `README.md` is a 9-byte stub (`# Errand`) → full clean-clone bring-up.
- `TEST_PLAN` does not exist → R1–R5 matrix + anti-cases + a ~2-min manual procedure.
- `BUGS.md` links only the P1 *recon* evidence → finalize by linking the P2–P4 **red-run** artifacts.
- Suite stability (3×), fresh-clone check, IP hygiene (zero APK/PDF in tree **and** history), ignore `.idea/`.

**Cardinal rule carries through every doc:** a test asserts the SPECIFICATION; a **red on a defect is
the correct result** — a finding, not a test bug. No assertion is ever bent to match the buggy app.

## Scope boundary (per lead, "nic poza tym")

**IN — repo-build (this build, Code):** README · TEST_PLAN · BUGS.md run-artifact linking ·
full-suite 3× stability · clean-machine / fresh-clone check · IP hygiene · `.gitignore` for `.idea/`.

**OUT — lead/Desktop-owned, executed in P5 *outside* the repo-build (NOT this build):**
- `WORKING-WITH-AI.md` consolidation — final English file generated by the lead, not Code.
- Orphaned-decision audit — sweep over the project conversations, on Desktop.
- `STRATEGIA-PROJEKT.md` section-10 sync — the file lives outside this repo.

→ In `PROGRESS.md` these three are labelled **"lead/Desktop-owned — done in P5 outside the
repo-build"** (English, per the all-files-English convention; lead's Polish intent rendered in English,
the P1-adjudication precedent). They are **NOT** marked deferred/out-of-scope.
→ **P5 is NOT marked fully DONE.** Only the repo-build slice is checked off. **Full P5 DONE =
repo-build (Code) + Desktop closeout (lead).**

**Everything else = README "Future work"** (considered-and-rejected, one-line why each).

**Hygiene constraint (binding):** requirements appear **only as paraphrase** — the source is the
already-paraphrased `docs/SPEC-MATRIX.md`. **Zero PDF content enters the repo.** Commits are local,
separate logical units; **push is the human's call.** No `Co-Authored-By` / "Generated with" trailers.

Code does **not** rule the stability verdict, does **not** rule any unexpected red, and does **not**
rule clean-machine pass. Code runs, reports real output verbatim, and **stops at each gate**.

- **Pattern expected from recon + P1 adjudication** (the *reference*, not a verdict Code issues):
  - **4 PASS** — `03-apply` (R2), `06-registration-validation` (R4), `07-registration-autologin` (R4), `08-login-logout` (R5).
  - **4 RED = findings** — `01-offers-locations` (R1/BUG-A), `02-offers-salary` (R1/BUG-C), `04-history-no-salary` (R3/BUG-B), `05-history-salary-format` (R3/BUG-C).
- **Report — then STOP. Hand the lead, do NOT proceed to B–G without an explicit OK:**

  - The **trimmed console** of **one** run (this is the only run that feeds the README "Example output").
  - **The stability verdict** (4 pass / 4 red, 0 flake) **and any unexpected red** = **the lead's
    adjudication**, not Code's. Code presents the three matrices; the lead rules stable / not-stable.
- **No churn (Correction 2):** the run regenerates `docs/runs/screens/` (01–05) and may write hierarchy,

  **`git checkout -- docs/runs/` to restore the ENTIRE directory.** Freshly-rendered screenshots may be

- **Self-validation guards:** paste **real** maestro output, never fabricate; if the emulator/app is not
  up, the run errors **before** asserting → **STOP and report** ("env not up"), do **not** narrate a
  result that didn't run. Precondition: ARM64 emulator running + `buggy.apk` installed (`adb install`).

### Task D — `README.md` (replace the stub) — sections per lead's enumeration
1. **Overview + cardinal rule** — what the suite is; tests assert the spec; a **red on a defect is the
   correct result** (a finding, not a test bug); the human issues BUG/PASS verdicts.
2. **What it tests / what it consciously does NOT** — covers R1–R5; deliberately out of scope:
   restart-persistence (spec silent → asserting it reads app behaviour), network (offline app, no
   `INTERNET` perm), and the BUG-B-on-list dedicated assertion (→ Known limitations).
3. **How to run — primary macOS path** — prerequisites = the **actually-installed** toolchain.
   **Correction 4:** before writing the prereqs, confirm the real versions with `java -version` and
   `maestro --version` (read-only) and write **those exact values**, never assumed numbers (don't write
   "JDK 21 / Maestro 2.6.1" unless the machine reports them). Plus Android **ARM64** emulator via `adb`,
   app installed. **`buggy.apk` is NOT in the repo** (company IP) → reviewer supplies it and `adb install`s
   it; then `maestro test .` → **expect 4 pass / 4 red**; note the run exits non-zero **by design**
   (4 findings). **Troubleshooting ABI:** must be `arm64-v8a` on M2; common emulator/`adb` gotchas
   (wrong ABI image, device not listed, app not installed).
4. **Architecture** — `flows/` (numbered 01–08 = the suite; `config.yaml` scopes the run) + `subflows/`
   (5 reusable fragments via `runFlow`: register-user, login, logout, open-profile, apply-to-offer;
   single-responsibility, each asserts only its readiness gate) + **selector conventions** (anchored
   DOTALL regex on the full `"label\nvalue"` node text; tap the clickable `Button` not the wrapper;
   `-e`/subflow-`env` precedence; `hideKeyboard` between fields; snackbar = transient → screenshot).
   Point to CLAUDE.md "Maestro / selector conventions" + `recon.yaml` comments as canonical (paraphrase,
   don't duplicate wholesale).
5. **Design decisions & trade-offs** — **Maestro** chosen (declarative YAML, fast bring-up, black-box
   against a **release** Flutter APK, no app source needed); **rejected Appium** (heavy driver/server +
   WebDriver boilerplate) and **Patrol** (needs app source / instrumentation — we have only a release
   APK). One-flow-per-finding isolation (Maestro aborts on first failed assertion → isolate R1/R3
   findings so one red never masks another). Subflows assert no outcome (caller owns the verdict).
6. **Known limitations** — **incl. the BUG-B-on-list documented gap**: Maestro's first-failure abort
   means flows 01/02 can't *also* assert the list `null` placeholder without masking their own BUG-A/C
   findings; the root defect is already **red in flow 04** (history surface) + captured as a
   **SPEC-MATRIX anti-case** + in `BUGS.md` → covered transitively, **no dedicated list flow** by design.
   Also: needs a live ARM64 emulator + a supplied APK; offline app; emulator-only (no physical-device run).
7. **Example output** — the trimmed real `maestro test .` console output from Task A (4 pass / 4 red).
8. **Future work (considered-and-rejected, one-line why each)** — CI (no hosted ARM64 emulator + offline
   app + APK not committable) · JUnit/report export · selective-run `tags:` · iOS · revisit Appium/Patrol
   · performance · visual / screenshot-diff regression.

  at Gate 1 for the lead's adjudication (do NOT self-rule it). Code's job is to *characterise* the cause on
  a real artifact (the P2/P3 harness-flake precedent) and present that evidence; never bend an assertion,
  never relabel a drive miss as a finding, never declare it a flake to clear the gate.
- **Don't narrate un-run results** — if the emulator/app isn't up, STOP and say so.
- **No scratch in commits** — confirm `git status` before every commit.
