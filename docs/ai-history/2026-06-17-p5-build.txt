       90 +                                                              
       91 +```                                                           
       92 +config.yaml            # Maestro workspace config ‚ÄĒ scopes `ma
          +estro test .` to flows/*.yaml only                            
       93 +flows/                 # the suite ‚ÄĒ 8 numbered flows, each = 
          +one requirement scenario                                      
       94 +  01-offers-locations.yaml      05-history-salary-format.yaml 
       95 +  02-offers-salary.yaml         06-registration-validation.yam
          +l                                                             
       96 +  03-apply.yaml                 07-registration-autologin.yaml
       97 +  04-history-no-salary.yaml     08-login-logout.yaml          
       98 +subflows/              # reusable fragments, invoked via `runF

       99 +  register-user.yaml  login.yaml  logout.yaml  open-profile.ya
          +ml  apply-to-offer.yaml

      103 +                                                              
      104 +- **Flows** are the suite. Each is self-contained: it sets up 
          +state (`clearState`, register a unique                        
      105 +  user), drives to the spec-defined moment, *proves the mechan
          +ic in green* (e.g. the list loaded, a                         
      106 +  clean tile renders), then makes the **spec assertion**. For 
          +the four findings, that last assertion is                     
      107 +  the red.                                                    
      108 +- **Subflows** are single-responsibility helpers reused across
          + flows. Each asserts only its own                             
      109 +  readiness gate and **never asserts an outcome** ‚ÄĒ the callin
          +g flow owns the verdict. `register-user`                      
      110 +  (params `EMAIL`/`PASSWORD`/`CONFIRM`), `apply-to-offer` (par
          +am `OFFER` = company anchor), `login`,                        
      111 +  `logout`, `open-profile`.                                   
      112 +                                                              
      113 +### Selector conventions (the load-bearing ones)              
      114 +                                                              
      115 +The full canonical list is in [`CLAUDE.md`](CLAUDE.md) ("Maest
          +ro / selector conventions") and the                           
      116 +per-screen map is in the comments of [`recon.yaml`](recon.yaml
          +). The essentials:                                            
      117 +                                                              
      118 +- **Selectors are regex anchored to the FULL node text (DOTALL
          +).** Flutter exposes text as a                                
      119 +  `"label\nvalue"` shape, so a bare substring does not match ‚ÄĒ
          + wrap tokens as `.*token.*` or use the                        
      120 +  exact full string.                                          
      121 +- **Tap the clickable `Button`, not the wrapper `View`** (the 
          +wrapper is often `clickable=false`).                          
      122 +- **Pass params only via the CLI `-e` / `runFlow` `env:` ‚ÄĒ dec
          +lare NO `env:` defaults.** In Maestro                         
      123 +  2.6.1 a flow/subflow `env` default *overrides* the value pas
          +sed in, silently blanking every field.                        
      124 +- **`hideKeyboard` between form fields.** Gboard has a transit
          +ional state that absorbs a tap fired                          
      125 +  mid-transition; dismissing the keyboard between fields makes
          + each tap land on a clean screen.                             
      126 +- **A snackbar is transient** (absent from the a11y hierarchy)
          + ‚Üí caught in-flow with                                        
      127 +  `extendedWaitUntil` on its token + a screenshot; the screens
          +hot is the evidence.

      129 +## 5. Design decisions & trade-offs                           
      130 +                                                              
      131 +- **Why Maestro.** Declarative YAML flows, fast bring-up, and 
          +it drives the app fully **black-box**                         
      132 +  against a **release** APK over `adb` ‚ÄĒ no app source, no ins
          +trumentation, no rebuild. That fits the                       
      133 +  brief exactly (we have an APK and a spec, not the source).  
      134 +- **Rejected Appium.** Heavier ‚ÄĒ a WebDriver server + client, 
          +language bindings, and capability/driver                      
      135 +  boilerplate ‚ÄĒ for no benefit here; the assertions we need ar
          +e straightforward UI presence/position                        
      136 +  checks Maestro expresses directly.                          
      137 +- **Rejected Patrol.** Patrol's strength is integration testin
          +g *from inside* the Flutter app, which                        
      138 +  needs the app source and a test harness build. We only have 
          +a release APK ‚Üí black-box is the only                         
      139 +  option, and Maestro covers it.                              
      140 +- **One flow per finding (isolation).** Maestro aborts a flow 
          +on its first failed assertion. If two                         
      141 +  findings shared a flow, the first red would mask the second.
          + So R1's two findings live in `01`/`02`                       
      142 +  and R3's two in `04`/`05`, each isolated, so every defect su
          +rfaces independently.                                         
      143 +- **Subflows assert no outcome.** A registration helper that a
          +sserted "success" could not be reused by                      
      144 +  the validation flow that expects an error. Keeping subflows 
          +outcome-neutral makes them composable;                        
      145 +  the caller asserts success or the specific error.
